1. Model Variants
gpt-5 FLAGSHIP
  • Complex reasoning & multi-step
  • Code generation & refactoring
  • Long context + tool calling
  • Highest capability, higher latency
gpt-5-mini
  • Cost-optimised reasoning & chat
  • Balanced speed, cost, capability
  • General chat, moderate tasks
  • Replace: o4-mini
gpt-5-nano
  • High-throughput simple tasks
  • Fastest, most cost-effective
  • Extraction, formatting, classify
  • Replace: gpt-4.1-nano

2. Responses API (Recommended)
Basic Call
from openai import OpenAI
client = OpenAI()

response = client.responses.create(
    model="gpt-5",
    input="Your prompt here",
    reasoning={"effort": "medium"},
    text={"verbosity": "medium"}
)
print(response.output_text)
Multi-turn with CoT Persistence
# First turn
first = client.responses.create(
    model="gpt-5",
    input="Design an LRU cache in Python",
    reasoning={"effort": "high"}
)

# Follow-up — reuses prior reasoning
follow_up = client.responses.create(
    model="gpt-5",
    input="Add thread-safety.",
    previous_response_id=first.id
)
Why Responses API over Chat Completions?
  • CoT persistence via previous_response_id
  • Fewer reasoning tokens re-generated
  • Higher cache hit rates
  • Lower latency on follow-ups
  • Improved cross-turn coherence
❌ Unsupported in GPT-5
  • temperature — not supported
  • top_p — not supported
  • logprobs — not supported

Use instead:

  • reasoning: {effort: ...}
  • text: {verbosity: ...}
  • max_output_tokens
3. Reasoning Effort Parameter
reasoning={"effort": "..."} — Default: "medium"
Value Speed Reasoning Tokens Best For
"minimal" NEW Fastest TTFT None / very few Extraction, formatting, simple classification, real-time UX
"low" Fast Light Quick responses, GPT-4.1 migrations
"medium" Balanced Moderate Default — general tasks, similar to o3
"high" Slowest Most Complex math, multi-step planning, difficult code
4. Verbosity Parameter
text={"verbosity": "..."} — Default: "medium"
Value Approx. Tokens Best For
"low" ~560 Terse UX, SQL queries, concise code, minimal prose
"medium" ~849 Balanced explanations (default)
"high" ~1288 Audits, teaching, hand-offs, detailed documentation
💡 Mixed Verbosity Strategy (Cursor pattern)

Set global verbosity="low" for prose, but instruct "use high verbosity for code tools" in the system prompt. Get concise explanations + detailed, well-commented code.

5. New Tool Features
Freeform / Custom Tool Calling (NEW)
# Raw text payloads — no JSON wrapping
response = client.responses.create(
    model="gpt-5",
    input="Use code_exec to find factorial of 10.",
    tools=[{
        "type": "custom",
        "name": "code_exec",
        "description": "Executes arbitrary Python code"
    }]
)
# Access: response.output[1].input  (raw code string)
# Feed back: {"type": "function_call_output",
#             "call_id": tool_call.call_id,
#             "output": "result_string"}
Context-Free Grammar (CFG) Constraints (NEW)
# Constrain output to exact syntax (Lark or Regex)
tools=[{
    "type": "custom",
    "name": "sql_tool",
    "description": "SQL tool — reason to conform",
    "format": {
        "type": "grammar",
        "syntax": "lark",  # or "regex"
        "definition": r"""
start: "SELECT" SP cols SP "FROM"
       SP table SP "WHERE" SP filter
cols: IDENTIFIER ("," SP IDENTIFIER)*
IDENTIFIER: /[A-Za-z_][A-Za-z0-9_]*/
SP: " "
"""
    }
}]
Freeform Tool — Key Points
  • Use for Python, SQL, shell, config — anything where JSON wrapping is unnatural
  • Model emits ResponseCustomToolCall with .input = raw string
  • ⚠ Does NOT support parallel tool calling
  • Great for: code sandboxes, SQL databases, shell envs
  • Feed result back via function_call_output to continue
CFG — Key Points & Best Practices
  • Lark: terminals (UPPER) matched first; rules (lower) combine them
  • Regex: provide pattern string directly
  • ✅ One bounded terminal for free-text between anchors
  • ❌ Don't split free-text across multiple terminals
  • Use SP = " " explicitly; avoid .* wildcards
  • If drifting OOD → tighten grammar + bump reasoning effort
Allowed Tools — Selective Access (NEW)
  • Define full toolkit, restrict active subset per request
  • Use for safety, predictability, preventing unintended calls
tool_choice={"type":"allowed_tools","mode":"auto","tools":[{"type":"function","name":"safe_tool"}]}
Tool Preambles
  • GPT-5 trained to give upfront plans + progress updates
  • Trigger: "Before calling a tool, explain why"
  • System prompt: "Rephrase goal, outline plan, narrate each step, summarize at end"
  • Useful for debugging and transparency

7. Prompting Best Practices
Controlling Agentic Eagerness

Less eager (quick, focused):

  • Start broad, fan out to focused subqueries in parallel
  • Stop as soon as you can name exact content to change
  • Early stop: top hits converge ~70% on one area
  • Batch search → minimal plan → complete task

More eager (autonomous, persistent):

  • "Keep going until query is completely resolved"
  • "Never stop when encountering uncertainty"
  • "Decide, proceed, document assumptions for user"
Instruction Following Rules
  • Keep instructions non-contradictory
  • Priority order: Security (CRITICAL) → Logic (HIGH) → Performance (MEDIUM) → Style (LOW)
  • Specify output format explicitly
  • Use structured system prompts with clear sections
  • Markdown not used by default — must prompt for it
  • Inline math: \( \), block math: \[ \]
Self-Reflection for High Quality

Add to system prompt:

  • Create a quality rubric with 5-7 categories internally
  • Evaluate and iterate until hitting top marks in ALL categories
  • Don't show rubric to user — it's for internal reasoning
  • Great for: one-shot app generation, frontend code, complex outputs
Metaprompting

Ask GPT-5 to improve its own prompts:

  • "Here's a prompt: {prompt}. Desired behavior: {X}. It instead {Y}. What minimal edits encourage improvement?"
  • Use reasoning.effort: "high" for metaprompting tasks
  • Keep as much of the original prompt intact as possible
  • Ask for specific phrases to add or remove

Retail Agent Pattern
  1. Authenticate user (email or name+zip)
  2. Retrieve relevant data
  3. Confirm before consequential actions
  4. Execute with reasoning: "minimal"
  5. One tool call at a time
SWE-Bench Pattern
  • Use reasoning: "high"
  • Verify changes extremely thoroughly
  • Account for hidden test edge cases
  • Make as many tool calls as needed
  • Triple-check before ending turn

9. Quick Reference & Gotchas
✅ Do This
  • Use Responses API (client.responses.create), not Chat Completions
  • Pass previous_response_id for multi-turn to save tokens + latency
  • Set reasoning.effort: "minimal" explicitly for speed-critical paths
  • Use text.verbosity: "high" for thorough code docs/audits
  • Keep CFG grammars simple — complex ones get API-rejected
  • Set parallel_tool_calls=False with custom tools
  • Use gpt-5-mini for follow-up processing of gpt-5 outputs
❌ Avoid This
  • Using temperature, top_p, logprobs — not supported in GPT-5
  • Parallel tool calling with custom/freeform tools — not supported
  • Splitting free-text across multiple CFG terminals — use one bounded terminal
  • Contradictory instructions in the same system prompt
  • Complex CFG grammars without testing at high reasoning effort first
  • Re-generating full CoT on every turn — use previous_response_id
  • Using Chat Completions for GPT-5 — missing CoT persistence
